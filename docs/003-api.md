# VeilleAnalytics — Contrat d'API du Worker

Référence des sept routes exposées par le Worker Cloudflare (`src/index.ts`), plus sa route de
repli. Le contrat décrit
ici est celui du code, pas une intention : chaque section renvoie au fichier qui l'implémente.

Les schémas de réponse sont donnés **sans valeurs de production** — les chiffres réels vivent
dans les exports datés, pas dans une documentation qui vieillirait mal. L'URL du Worker s'écrit
`https://<worker>` partout.

## 1. Deux autorisations, deux porteurs

Ingérer un article et lire la base ne sont pas le même droit et n'ont pas le même porteur. Les
deux jetons vivent dans le même namespace KV `AUTH` (cf. [README](../README.md#jetons-kv-auth)),
mais sont portés par des en-têtes distincts pour qu'aucune confusion ne soit possible.

| Droit | En-tête | Clé KV | Porteur |
|---|---|---|---|
| Ingérer | `Authorization: Bearer <jeton>` | `API_TOKEN` | le flux Node-RED |
| Lire | `X-Dashboard-Token: <jeton>` | `READ_TOKEN` | le proxy Nitro du dashboard |

**Le refus est l'état par défaut en cas de panne.** Si `READ_TOKEN` est absent du KV, la requête
est refusée plutôt qu'acceptée (`src/index.ts`, `verifieJetonLecture`) : une erreur de
configuration doit fermer, jamais ouvrir silencieusement.

Les deux comparaisons passent par `jetonsEgaux` (`src/lib/jetons.ts`), en **temps constant** :
`!==` sur des chaînes s'arrête au premier octet qui diffère, ce qui laisse reconstruire un jeton
octet par octet. La longueur du jeton attendu, elle, reste observable — limite énoncée à
l'ADR D18 plutôt que masquée.

### Whitelist des lectures protégées

Cinq chemins sont protégés, énumérés dans la constante `LECTURES_PROTEGEES` :

```
/api/articles
/api/stats/themes
/api/stats/sources
/api/stats/timeline
/api/stats/ml-comparison
```

Le choix d'une **whitelist** plutôt que d'une blacklist est assumé et documenté en commentaire :
une route absente de cette liste est publique, donc une route ajoutée demain sans y penser sera
publique. La liste inverse ferait passer pour protégée une route qu'on aurait simplement oublié
d'inscrire — le défaut serait alors invisible. Deux exceptions volontaires demeurent :
`/api/stats/health`, appelée hors session pour la supervision, et la route de repli ; ni l'une ni
l'autre ne rend de donnée d'article.

### En-têtes de sécurité

Ils sont appliqués à **toutes** les réponses, y compris les 401, 403 et 500, en un seul point à
la sortie du routeur (`withSecurityHeaders`, `src/lib/securityHeaders.ts`) plutôt qu'à chaque
construction de réponse :

```
X-Content-Type-Options: nosniff
Content-Security-Policy: default-src 'none'; frame-ancestors 'none'
Referrer-Policy: no-referrer
Strict-Transport-Security: max-age=63072000; includeSubDomains
```

Le jeu est volontairement plus court que celui du dashboard : une API JSON ne rend pas de
document, ni CSP de rendu ni `Permissions-Policy` n'y ont d'objet.

## 2. `POST /api/ingest`

Ingestion d'un article, appelée par le flux Node-RED après classification Mistral.

```http
POST /api/ingest
Authorization: Bearer <API_TOKEN>
Content-Type: application/json

{
  "title": "Titre de l'article",        // obligatoire
  "link": "https://…",                  // obligatoire, contrainte UNIQUE en base
  "source": "nom_source",               // défaut : "inconnu"
  "resume": "…",
  "categorie": "PRO",
  "score": 3,
  "date": "Tue, 21 Jul 2026 08:00:00 GMT",
  "tags": ["tag1", "tag2"],
  "themes": ["Architecture", "DevOps/Infrastructure"]
}
```

| Code | Cas |
|---|---|
| `201` | article inséré — corps `{ "status": "ok" }` |
| `400` | JSON illisible, ou `title` / `link` manquant |
| `401` | en-tête `Authorization` absent ou non `Bearer` |
| `403` | jeton présent mais différent de `API_TOKEN` |
| `500` | erreur D1 — corps `{ "error": "…" }` |

Deux détails de contrat méritent d'être connus des appelants.

**`201` ne signifie pas « nouvel article ».** L'insertion est un `INSERT OR IGNORE` sur la
contrainte `url UNIQUE` : renvoyer un article déjà connu renvoie `201` sans rien écrire.
L'idempotence est voulue — c'est elle qui rend la collecte rejouable — mais elle interdit de
compter les articles nouveaux à partir des codes de retour.

**Deux effets de bord suivent une insertion réelle** (`meta.changes > 0`), et seulement dans ce
cas :

- la **classification ML** part en arrière-plan via `ctx.waitUntil` : la réponse `201` n'attend
  pas Hugging Face. Un échec laisse `themes_ml` à `NULL`, état rattrapable par
  `scripts/classify-ml.mjs` et signalé par `/api/stats/health` ;
- l'**agrégat du jour** est rafraîchi de façon **synchrone** si l'article porte une date lisible
  (ADR D11) : `/api/stats/timeline` le lit directement, il doit donc être exact dès le retour.
  Un échec de rafraîchissement est journalisé sans faire échouer l'ingestion, et se rattrape par
  `scripts/rebuild-aggregates.sql`.

Les dates entrantes sont normalisées en ISO 8601 à l'ingestion (`toIsoOrNull`, ADR D09) : une
date illisible devient `NULL` plutôt que de propager une chaîne RFC 822 dans la base. Les tags
sont mis en minuscules et détourés (`normalizeTags`).

## 3. `GET /api/articles`

Liste paginée et filtrée. Requiert `X-Dashboard-Token`.

### Paramètres de requête

| Paramètre | Type | Défaut | Effet |
|---|---|---|---|
| `page` | entier ≥ 1 | `1` | numéro de page |
| `limit` | entier 1-100 | `20` | taille de page, bornée en dur |
| `theme` | chaîne | — | l'article porte ce thème dans `themes_mistral` |
| `source` | chaîne | — | égalité stricte sur `source` |
| `categorie` | chaîne | — | égalité stricte sur `categorie_mistral` |
| `score_min` | entier | — | `score_mistral >= n` |
| `theme_ml` | chaîne | — | l'article porte ce thème dans `themes_ml` |
| `score_ml_min` | décimal | — | `score_confiance_ml >= n` |
| `ml` | `oui` \| `non` | — | présence (`oui`) ou absence (`non`) d'une classification ML |
| `desaccord` | `1` | — | voir ci-dessous |

Les filtres se combinent en `AND`. Une valeur non numérique sur `score_min` ou `score_ml_min` est
ignorée plutôt que rejetée ; `page` et `limit` sont ramenés dans leurs bornes.

**`desaccord=1` a un sens précis** : les deux classifications existent **et** n'ont **aucun**
thème en commun. Un recouvrement partiel n'est donc pas un désaccord, et un article non classifié
par l'un des deux systèmes n'en est pas un non plus — il est simplement hors comparaison. C'est
la même convention que `/api/stats/ml-comparison`.

Les sept thèmes canoniques attendus par `theme` et `theme_ml` sont : `IA/ML`,
`DevOps/Infrastructure`, `Architecture`, `Sécurité`, `Développement`, `Pratiques/Qualité`,
`Productivité/Outils` (`LABEL_MAP`, `src/lib/classifyMl.ts`).

### Réponse

```jsonc
{
  "data": [
    {
      "id": 1,
      "titre": "…",
      "url": "https://…",
      "resume": "…",
      "source": "…",
      "categorie_mistral": "PRO",
      "score_mistral": 3,
      "themes_mistral": ["Architecture"],   // tableau ; [] si jamais classifié
      "themes_ml": ["Architecture"],        // null = jamais classifié ; [] = classifié, aucun thème au seuil
      "score_confiance_ml": 0.83,           // score du thème le mieux classé, même sous le seuil
      "tags": ["tag1"],                     // tableau
      "date_article": "2026-07-21T08:00:00.000Z",
      "date_collecte": "2026-07-21T09:12:04.512Z"
    }
  ],
  "pagination": { "page": 1, "limit": 20, "total": 0, "totalPages": 0 }
}
```

`themes_mistral`, `themes_ml` et `tags` sont stockés en TEXT JSON et **désérialisés à la
sortie** (`parseArticleRow`) : le client reçoit des tableaux, pas des chaînes.

La distinction **`null` ≠ `[]`** sur `themes_ml` porte du sens et ne doit pas être aplatie :
`null` signifie « jamais classifié » — un échec ou une classification en attente — tandis que `[]`
signifie « classifié, mais aucun thème n'a dépassé le seuil de 0,7 ». Les deux appellent des
traitements opposés, l'un un rattrapage, l'autre rien.

Tri : `date_article DESC`, puis `date_collecte DESC` pour départager les articles sans date.

## 4. `GET /api/stats/themes` et `GET /api/stats/sources`

Comptages simples, protégés par `X-Dashboard-Token`.

```jsonc
// GET /api/stats/themes — un article multi-thèmes compte dans chacun de ses thèmes
{ "data": [ { "theme": "Architecture", "count": 0 } ] }

// GET /api/stats/sources
{ "data": [ { "source": "nom_source", "count": 0 } ] }
```

Sur `themes`, la somme des `count` dépasse donc le nombre d'articles : le dépliage par `json_each`
compte un article une fois par thème porté. Pour un total non double-compté, utiliser la ligne de
rollup de `timeline`.

---

## 5. `GET /api/stats/timeline`

Volume d'articles par jour. Protégé par `X-Dashboard-Token`.

```jsonc
{ "data": [ { "jour": "2026-07-21", "count": 0 } ] }
```

La route **lit l'agrégat pré-calculé** `agg_quotidien` sur ses lignes de rollup
(`thematique IS NULL`) au lieu d'un `GROUP BY` à la volée sur `articles` — c'est le principe même
de l'entrepôt décisionnel, l'agrégat étant maintenu à l'écriture par l'ingestion (ADR D11). Le
contrat de sortie est resté inchangé lors de cette bascule.

Le recours à la ligne de rollup n'est pas un détail : sommer les lignes par thème
double-compterait les articles multi-thèmes. La commande de contrôle croisé qui vérifie la
concordance entre l'agrégat et le calcul direct est dans le [README](../README.md) — elle doit
renvoyer zéro ligne.

## 6. `GET /api/stats/ml-comparison`

Concordance entre la classification Mistral et la classification ML. Protégé par
`X-Dashboard-Token`. Schéma : interface `MlComparison`, `src/lib/mlComparison.ts`.

```jsonc
{
  "data": {
    "global": {
      "total": 0,          // articles en base
      "compares": 0,       // articles portant LES DEUX classifications
      "accord_exact": 0,   // ensembles de thèmes strictement identiques
      "chevauchement": 0,  // au moins un thème commun
      "jaccard_moyen": 0   // moyenne des |∩|/|∪|, arrondie au millième
    },
    "par_theme": [
      { "theme": "Architecture", "accord": 0, "mistral_seul": 0, "ml_seul": 0 }
    ]
  }
}
```

**Seuls les articles portant les deux classifications sont comparés** : `compares` est donc
inférieur à `total`, et c'est `compares` — non `total` — qui est le dénominateur de toute lecture
des trois compteurs. Deux ensembles vides sont tenus pour identiques (Jaccard = 1 par convention).

Le calcul est fait en TypeScript plutôt qu'en SQL : la logique ensembliste est plus lisible et
testable unitairement qu'en `json_each` imbriqués, pour un volume qui ne le justifie pas.

Un thème hors référentiel rencontré dans les données est ignoré du décompte `par_theme` — cas
réel en production, quelques libellés fautifs subsistent en base et sont assumés comme anomalie
qualité documentée plutôt que corrigés silencieusement.

## 7. `GET /api/stats/health`

Santé du pipeline. **Volontairement non protégée** : la supervision s'appelle hors session, et la
route ne rend aucune donnée d'article. Schéma : interface `PipelineHealth`, `src/lib/health.ts`.

```jsonc
{
  "data": {
    "statut": "ok",                     // le pire des deux statuts ci-dessous
    "collecte": {
      "dernier_article_collecte": "2026-07-21T09:12:04.512Z",
      "jours_depuis": 0,                // null si aucune date exploitable
      "statut": "ok"
    },
    "classification": {
      "total": 0,
      "ml_en_retard": 0,                // themes_ml NULL et collecté il y a plus de 24 h
      "ml_sans_theme": 0,               // themes_ml = [] : classifié, rien au-dessus du seuil
      "mistral_manquants": 0,
      "statut": "ok"
    }
  }
}
```

Statuts possibles : `ok`, `degrade`, `alerte`. Le statut global est le pire des deux.

| Constante | Valeur | Rôle |
|---|---|---|
| `FRAICHEUR_OK_JOURS` | 3 | au-delà, la collecte passe `degrade` |
| `FRAICHEUR_ALERTE_JOURS` | 14 | au-delà, la collecte passe `alerte` |
| `ML_RETARD_HEURES` | 24 | au-delà, un `themes_ml` encore `NULL` est un échec avéré |

Les seuils de fraîcheur sont ancrés sur le **meilleur régime réellement atteint**, pas sur le
régime observé aujourd'hui : les calibrer sur la dérive constatée reviendrait à normaliser la
déviance et à produire un voyant vert par construction (ADR D12).

**Seul `ml_en_retard` pilote le statut de classification.** `ml_sans_theme` et
`mistral_manquants` sont exposés mais neutres, et le choix est délibéré : le premier n'est pas un
échec — l'article a bien été classifié, aucun thème n'a simplement franchi le seuil — et le second
est un résidu historique figé de la migration initiale. Les faire remonter en alerte rendrait
l'indicateur définitivement rouge, donc inutile.

Deux limites sont énoncées plutôt que contournées. Le repère de fraîcheur est le dernier article
**inséré**, pas la dernière passe de collecte : le pipeline ne journalise pas ses passes, et une
passe ne ramenant que des doublons n'écrit rien. Et `ml_en_retard` n'offre aucun moyen
d'acquitter un article définitivement non classifiable — un seul contenu durablement refusé par
Hugging Face épinglerait le statut à `alerte` sans recours ; le cas ne se manifeste pas
aujourd'hui, la limite est tracée à l'ADR D12.

> ⚠️ Les comptages de cette route sont **dupliqués** dans `scripts/health-check.sql`, version
> exploitation à lancer en `--remote` pour recouper l'endpoint. Toute évolution va aux deux
> endroits. Les deux ne comptent d'ailleurs pas de la même façon — comparaison de chaînes ISO
> ici, arithmétique `julianday` là-bas — et ne coïncident que si tout `date_collecte` a la forme
> d'un `toISOString()`, condition mesurée en production plutôt que supposée.

## 8. Route de repli

Toute autre requête, quelle que soit la méthode, renvoie `200` avec le corps texte
`VeilleAnalytics API - OK`. C'est une surface publique assumée : elle ne rend aucune donnée et
sert de sonde de disponibilité. Elle est, avec `/api/stats/health`, la seconde exception
volontaire à la whitelist.

Un point de comportement à connaître : la vérification du jeton de lecture porte sur le
**chemin**, indépendamment de la méthode. Un `POST /api/articles` est donc refusé en `401` sans
jeton, mais retombe sur cette route de repli — et répond `200` — avec un jeton valide. Il n'y a
pas de `405` dans ce contrat.

## 9. Vérifier le contrat

Les tests d'intégration tournent sur une D1 Miniflare réelle chargée avec le schéma de
production (`@cloudflare/vitest-pool-workers`) : ils exercent les routes, pas des doublures.

```bash
pnpm test          # 140 tests / 7 fichiers (vérifié le 2026-07-23)
```

Relevé manuel du contrôle d'accès, la preuve étant la **réponse HTTP** et non le code :

```bash
curl -i https://<worker>/api/articles                          # 401 sans en-tête
curl -i -H "X-Dashboard-Token: <READ_TOKEN>" https://<worker>/api/articles   # 200
curl -i https://<worker>/api/stats/health                      # 200, non protégée
```
